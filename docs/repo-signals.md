## Architectural Patterns

### Serverless Event-Driven Architecture

**Files:** `src/orchestrator/trigger_handler.py`, `worker_handler.py`, `status_handler.py`, `infra/template.yaml`

API Gateway → Lambda → Step Functions → Lambda (worker) → Bedrock Agents. Every run is async; clients poll `GET /team/task/{run_id}` for terminal state.

---

### Pipeline / Data-Flow Pattern

**Files:** `src/orchestrator/worker_handler.py`, `handler.py`

Linear agent pipeline where the output of step N is injected as input to step N+1. Results accumulate in an `outputs` dict that every downstream step can read.

---

### Multi-Agent Collaboration

**Files:** `src/orchestrator/worker_handler.py`, `handler.py`, `infra/bedrock-agents.yaml`

Multiple Amazon Bedrock agents collaborate sequentially. A supervisor agent (`role="supervisor"`) produces a brief that guides specialist agents (strategist, writer, editor, distributor).

---

### Configuration-Driven Workflow

**Files:** `src/orchestrator/config_loader.py`, `config/examples/teams/`

All workflow logic lives in JSON stored in S3 (`team.json`): agents, pipeline steps, RAG settings, output schemas, and feature flags. No Lambda code changes are needed to create or modify workflows.

---

### Step Functions Orchestration

**Files:** `src/orchestrator/trigger_handler.py`, `status_handler.py`, `infra/template.yaml`

Trigger Lambda calls `start_execution()`; Status Lambda polls `describe_execution()`. Worker Lambda is the activity processor invoked by the state machine for each pipeline run.

---

### Async Request / Poll Pattern

**Files:** `src/orchestrator/trigger_handler.py`, `status_handler.py`

`POST /team/task` returns `run_id` immediately. The caller polls `GET /team/task/{run_id}` until `SUCCEEDED` or `FAILED`.

---

### Infrastructure as Code (SAM / CloudFormation)

**Files:** `infra/template.yaml`, `infra/bedrock-agents.yaml`, `infra/samconfig.toml`

AWS SAM templates parameterise all resources (VPC, buckets, DynamoDB, Bedrock agent IDs). Conditional blocks create or reuse resources based on parameters (`CreateConfigBucket`, `CreateDdbTable`, `DeployWAF`).

---

### VPC-Isolated Lambda

**Files:** `infra/template.yaml`, `src/orchestrator/rag.py`

Lambda functions run inside a private VPC with ENIs so they can reach RDS (pgvector). Security-group ingress rules are generated by SAM.

## Structural Patterns

### Data Access Object (DAO)

**Files:** `src/orchestrator/db.py`

`DbDao` encapsulates all DynamoDB operations: `get_item`, `put_item`, `put_run_meta`, `put_step`, `list_tasks`, `mark_task_done`, `list_completed_topic_levels`. `DbDao.from_team_config()` is the preferred factory constructor.

---

### Lambda Handler / Adapter

**Files:** `src/orchestrator/handler.py`, `trigger_handler.py`, `worker_handler.py`, `status_handler.py`

Each handler normalises the API Gateway event via helpers (`_method`, `_path`, `_qs`, `_json_body`, `_cors`, `_resp`) then delegates to pure business-logic functions, keeping HTTP concerns separate from domain logic.

---

### Factory Method

**Files:** `src/orchestrator/config_loader.py`, `db.py`, `models.py`

`load_team_config()` returns a typed `(TeamConfig, raw_dict)` tuple. `DbDao.from_team_config()` builds a DAO from config. Dataclass constructors (`TeamGlobals`, `AgentConfig`, `BedrockRef`) build typed objects.

---

### Facade

**Files:** `src/orchestrator/storage.py`

`save_artifact()` is a thin facade over S3: it constructs the key path (`runs/{run_id}/{step_id}.json`), serialises to JSON, and calls `put_object`.

---

### Decorator / Wrapper (Telemetry)

**Files:** `src/orchestrator/bedrock_invoke.py`, `bedrock_wrappers.py`, `mcp_observatory.py`

`mcp_observatory` wraps every Bedrock invocation with `observe_agent_request` / `observe_model_request`, injecting latency, token counts, cost, and optional shadow-model comparison without modifying core invocation code.

---

### Builder / Fluent Assembly

**Files:** `src/orchestrator/prompt_builder.py`, `enrich.py`

`build_prompt()` assembles multi-section LLM prompts by conditionally appending RAG context, owner profile, Gemini brief, schema instructions, and prior step outputs.

---

### Strategy (RAG Mode Selection)

**Files:** `src/orchestrator/rag.py`, `profile_context.py`, `gemini.py`

`get_rag_context()` dispatches to different retrieval strategies at runtime based on `rag.mode` in team config: `"explicit"` (pgvector cosine search), `"history"` (DynamoDB execution history), or `"kb"` (Bedrock Knowledge Base).

---

### Template Method

**Files:** `src/orchestrator/worker_handler.py`, `handler.py`

`run_team_pipeline()` defines the invariant workflow skeleton (load config → init DAO → get RAG context → loop steps → invoke agent → validate → save artifact → mark done). Individual steps are pluggable via config.

---

### Transformer / Converter

**Files:** `src/orchestrator/structured_transform.py`, `json_utils.py`, `schema_validate.py`

`transform_json_to_schema()` uses Claude to reshape agent JSON output to a target schema. `extract_json_payload()` applies multi-strategy fuzzy extraction. `normalize_target_schema()` converts JSON Schema ↔ template.

---

## Behavioural Patterns

### Chain of Responsibility (Step Pipeline)

**Files:** `src/orchestrator/worker_handler.py`, `handler.py`

Each pipeline step receives the accumulated `outputs` dict from all previous steps. `_build_step_inputs()` merges them so any downstream step can access any predecessor's result. The chain short-circuits on `StepFailed`.

---

### Retry with Exponential Back-off

**Files:** `src/orchestrator/bedrock_invoke.py`

`invoke_agent()` retries up to `max_retries` (default 2) on transient errors. Auth / token errors raise `StepFailed` immediately (fail-fast). Backoff delay doubles between attempts.

---

### Fallback / Graceful Degradation

**Files:** `src/orchestrator/json_utils.py`, `enrich.py`, `structured_transform.py`

`extract_json_payload()` cascades through four extraction strategies before giving up. `enrich_step_output()` returns `raw_output` on any exception. `_build_transform_fallback()` coerces failed LLM transformations to a safe default response.

---

### Observer / Telemetry

**Files:** `src/orchestrator/mcp_observatory.py`, `amp_metrics.py`

Invocation wrappers emit metrics to DynamoDB (`ObservatoryMetricsTable`) and push Protobuf-encoded time-series to Amazon Managed Prometheus via `remote_write` — decoupled from core pipeline logic.

---

### Dual Inference / Shadow Model

**Files:** `src/orchestrator/mcp_observatory.py`, `bedrock_invoke.py`

If `shadow_alias_id` is present, a second model alias runs in parallel via `asyncio`. Telemetry records `shadow_disagreement_score` and `shadow_numeric_variance` for A/B model evaluation.

---

### Enrichment / Post-Processing

**Files:** `src/orchestrator/enrich.py`

`enrich_step_output()` uses Claude to detect placeholder / refusal text from Nova, rewrite copy to the owner's voice, enforce schema compliance, and regenerate from step context when needed.

---

### Guardrail Intercept

**Files:** `src/orchestrator/bedrock_invoke.py`

Bedrock `INTERVENED` guardrail actions are detected in response headers, logged, and tracked separately so they can be audited without failing the step silently.

---

### Custom Exception Domain Model

**Files:** `src/orchestrator/models.py`

`StepFailed(step_id, message, raw_output)` carries structured context about a failed workflow step, caught at the handler boundary and serialised into a standard error response.

---

## Data / Persistence Patterns

### Composite Key (DynamoDB)

**Files:** `src/orchestrator/db.py`

Partition + sort key conventions:

| Entity | pk | sk |
|---|---|---|
| Run metadata | `RUN#{run_id}` | `META` |
| Run step | `RUN#{run_id}` | `STEP#{step_id}` |
| Task | `TASK#{owner}` | `DATE#{ymd}#TASK#{task_id}` |
| Observatory | `OBSERVATORY#{operation}` | `{ts}#{trace_id}` |

---

### TTL / Auto-Expiration

**Files:** `src/orchestrator/mcp_observatory.py`

Observatory metric items carry a DynamoDB TTL of 90 days for automatic cleanup without manual purge jobs.

---

### S3 Artifact Lifecycle

**Files:** `infra/template.yaml`, `src/orchestrator/storage.py`

Step output JSON is stored at `s3://{bucket}/runs/{run_id}/{step_id}.json`. S3 lifecycle rules transition objects to IA after 30 days, GLACIER after 90 days, and expire them after 365 days.

---

### Caching / Lazy Initialisation

**Files:** `src/orchestrator/rag.py`, `mcp_observatory.py`, `enrich.py`

Module-level singletons (`_db_secret_cache`, `_ddb_table`, `_bedrock_runtime`) are populated on first use to amortise Lambda cold-start overhead across warm invocations.

---

## Validation Patterns

### JSON Schema Validation

**Files:** `src/orchestrator/schema_validate.py`, `config/examples/schemas/`

Agent outputs are validated against JSON Schema Draft 2023-12 via the `jsonschema` library. `validate_or_unwrap_output()` also handles single-key envelope responses and schema-specific normalisations (e.g. `objective` → `goal` for `CreativeBriefV1`).

---

### Type Coercion

**Files:** `src/orchestrator/structured_transform.py`

`_coerce_to_template()` applies best-effort type conversion: empty string fallback for strings, `0` for numeric parse errors, case-insensitive bool parsing (`"true"` / `"1"` / `"yes"`), and recursive coercion for objects/arrays.

---

### Output Normalisation

**Files:** `src/orchestrator/json_utils.py`, `structured_transform.py`, `enrich.py`

Pipeline strips markdown fences, removes control characters, normalises curly quotes to straight quotes, and recursively decodes double-encoded JSON (max depth 3) before validation.

---

### PII Redaction

**Files:** `src/orchestrator/profile_context.py`

Regex patterns scrub emails, SSNs, and phone numbers from RAG context before injecting it into LLM prompts.

---

## Integration Patterns

### Secrets Manager Credential Injection

**Files:** `src/orchestrator/rag.py`, `gemini.py`, `mcp_observatory.py`, `infra/template.yaml`

All credentials (DB password, Gemini API key) are stored in AWS Secrets Manager. ARNs are injected as environment variables by SAM; Lambda fetches the secret value at runtime. Never hardcoded.

---

### Environment Variable Configuration

**Files:** `infra/template.yaml`, `src/orchestrator/trigger_handler.py`, `worker_handler.py`, `status_handler.py`

SAM wires CloudFormation resource ARNs/names into Lambda environment variables (`CONFIG_BUCKET`, `ARTIFACT_BUCKET`, `DDB_TABLE`, `STATE_MACHINE_ARN`, etc.). Runtime code reads them via `os.environ`.

---

### Protobuf Remote-Write (AMP)

**Files:** `src/orchestrator/amp_metrics.py`

Hand-coded varint/wire-type Protobuf encoder (no grpc dependency) builds a `WriteRequest` and pushes time-series to Amazon Managed Prometheus via the Prometheus `remote_write` HTTP endpoint with Snappy compression.

---

### pgvector Similarity Search

**Files:** `src/orchestrator/rag.py`

Text is embedded with Bedrock Titan; the resulting vector is used for cosine distance search (`<=>`) against the `rag_chunks` table in RDS PostgreSQL 15 + pgvector. Falls back to `ILIKE` keyword search if embedding is unavailable.

---

### Gemini External Research

**Files:** `src/orchestrator/gemini.py`

Google Gemini 2.0 Flash is called as an optional enrichment step to generate a research brief that is injected into the pipeline context (`features.gemini_research` flag in team config).

---

### Conditional Feature Flags (Config)

**Files:** `config/examples/teams/`, `src/orchestrator/worker_handler.py`

`features.gemini_research`, `features.explicit_rag`, `rag.mode`, and `owner_profile.enabled` in `team.json` toggle capabilities at runtime without code changes.
